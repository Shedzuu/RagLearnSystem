## Архитектурное решение системы

### 1. Общая идея системы

Система представляет собой **AI‑поддерживаемую учебную платформу** (LMS), где пользователь:

- загружает свой учебный материал (книгу, конспект и т.п.);
- выбирает интересующие темы;
- получает автоматически сгенерированный **курс (план обучения)** с разделами, уроками, теорией и заданиями;
- проходит курс, получает **фидбэк от LLM** и аналитику по слабым местам;
- может общаться с **AI‑чатом‑репетитором**, который отвечает в контексте конкретного курса, раздела, урока или вопроса.

Технологический стек:

- **Backend**: Django + Django REST Framework + PostgreSQL, JWT‑аутентификация.
- **Frontend**: React (Vite), SPA с маршрутами под страницы лендинга, авторизации и работы с курсами.
- **AI‑слой**: LLM + векторное хранилище (RAG) — для генерации структуры курса и контекстных ответов.

---

### 2. Слои доменной модели

Архитектура домена разделена на **три логических слоя**:

1. **Контент (статический слой)** — то, что создаётся один раз при генерации курса и не меняется при прохождении:
  - `plans_plan` → `plans_section` → `plans_unit` → `plans_question` → `plans_choice`.
2. **Действия пользователя (динамический слой)** — история взаимодействия пользователя с курсом:
  - `learning_enrollment` → `learning_attempt` → `learning_answer` → `learning_answer_choice`.
3. **Агрегированная аналитика и фидбэк** — быстрый доступ к прогрессу и слабым местам:
  - `learning_unit_progress`, `learning_section_progress`, `learning_question_stats`, `learning_feedback`, `learning_answer.feedback_text`.

Отдельно вынесен **слой AI‑чата**:

- `ai_chat_message` — история диалогов с привязкой к плану/разделу/уроку/вопросу.

---

### 3. Модель контента (курс)

#### 3.1 Пользователи

- `users_user` — кастомная модель пользователя (наследник `AbstractUser`):
  - логин по e‑mail (`USERNAME_FIELD = 'email'`);
  - совместимость с экосистемой Django (админка, права, аутентификация).

#### 3.2 План (курс) и статусы генерации

- `plans_plan`:
  - `owner_id` — автор/владелец курса;
  - `title`, `description` — метаданные;
  - `generation_status` — состояние общей RAG‑генерации плана (`pending / processing / ready / failed`);
  - `is_public` — флаг видимости;
  - временные метки `created_at`, `updated_at`.

Решение: **генерация курса асинхронна** по отношению к пользователю. Пока статус `processing`, план считается в работе, но за счёт инкрементальной генерации (см. ниже) отдельные разделы могут становиться доступны раньше, чем весь план будет полностью готов.

#### 3.3 Исходный документ

- `plans_document`:
  - связь одного или нескольких файлов с конкретным планом (`plan_id`);
  - хранит путь к файлу, оригинальное имя, размер, дату загрузки.

Документы индексируются в векторном хранилище, но сами эмбеддинги **не хранятся в реляционной БД**: идентификация чанков делается через метаданные (`plan_id`, `section_id`, `unit_id`) в векторном индексе.

#### 3.4 Структура плана: разделы и уроки

- `plans_section`:
  - FK на `plans_plan`;
  - `title`, `order` — название раздела и его позиция (Раздел 1, 2, 3…);
  - `generation_status` — состояние генерации содержимого раздела (`draft / generating / ready / failed`);
  - `UNIQUE(plan_id, order)` обеспечивает стабильный порядок.
- `plans_unit`:
  - FK на `plans_section`;
  - `title`, `order` — заголовок и порядок внутри раздела (1.1, 1.2 и т.п.);
  - `theory` — текст теории (конспект урока для пользователя);
  - `generation_status` — состояние генерации конкретного урока (`draft / generating / ready / failed`).

Решение: **Unit — минимальная учебная единица («экран урока»)**, которая объединяет теорию и набор вопросов. Глубже вкладывать уровни (1.1.1 и т.п.) не требуется для дипломного проекта.

#### 3.5 Вопросы и варианты

- `plans_question`:
  - FK на `plans_unit`;
  - `text` — формулировка вопроса;
  - `type` — тип вопроса (`single_choice`, `multiple_choice`, `open_text`, `code`);
  - `difficulty` — уровень сложности (1–3);
  - `order` — порядок отображения внутри юнита;
  - `points` — базовый вес вопроса.
- `plans_choice`:
  - FK на `plans_question`;
  - `text` — вариант ответа;
  - `is_correct` — флаг правильности (для тестов);
  - `order` — порядок вариантов.

Решение: **логика проверки тестов вынесена на уровень БД** через `is_correct` и `points`. LLM используется для генерации вопросов и вариантов, но проверка в рантайме может быть чисто детерминированной.

### 4. Прохождение курса и прогресс

#### 4.1 Запись на курс

- `learning_enrollment`:
  - `user_id`, `plan_id` — уникальная пара (один пользователь — одна запись на курс);
  - `status` — состояние прохождения (`active / completed / dropped`);
  - `enrolled_at` — дата начала.

Даже если пользователь создаёт курс сам для себя, такое разделение:

- позволяет масштабировать модель на много пользователей;
- упрощает аналитику по каждому прохождению.

#### 4.2 Попытки и ответы

- `learning_attempt`:
  - FK на `learning_enrollment`;
  - временные метки начала и завершения;
  - итоговый `%` за попытку (`score_percent`).
- `learning_answer`:
  - FK на `learning_attempt` и `plans_question`;
  - `text_answer`, `code_answer` — содержимое ответа;
  - `is_correct`, `earned_points` — результат проверки;
  - `feedback_text` — текстовый фидбэк от LLM (объяснение ошибки или подсказка);
  - уникальность по (`attempt_id`, `question_id`).
- `learning_answer_choice`:
  - связь many‑to‑many между ответом и выбранными вариантами (`plans_choice`);
  - подходит и для одного, и для нескольких правильных ответов.

Решение: хранить **сырые ответы и фидбэк** в явном виде, чтобы:

- можно было пересчитать прогресс при изменении правил оценивания;
- использовать историю ответов в AI‑чате.

#### 4.3 Прогресс по урокам и разделам

- `learning_unit_progress`:
  - FK на `learning_enrollment` и `plans_unit`;
  - `completion_percent`, `is_completed`, `last_updated`.
- `learning_section_progress`:
  - аналогичная структура для `plans_section`.

Решение: **агрегаты по прогрессу хранятся отдельно** от сырых ответов, чтобы:

- не делать тяжёлые агрегации по `learning_answer` на каждый запрос;
- быстро выводить прогресс по курсу/разделам на фронтенде.

#### 4.4 Аналитика слабых мест и общий фидбэк

- `learning_question_stats`:
  - количество попыток и успешных ответов по каждому вопросу;
  - `success_rate` и `last_attempt_at`.
- `learning_feedback`:
  - итоговый отзыв пользователя по курсу (`rating`, `comment`).

На основе этих таблиц можно:

- выделять темы/юниты, где успехи ниже порога;
- формировать **AI‑отчёт о слабых сторонах** по завершении курса;
- рекомендовать дополнительные материалы или повторение конкретных уроков.

---

### 5. AI‑чат и контекстные ответы

#### 5.1 История чата

- `ai_chat_message`:
  - `user_id` — автор сообщения;
  - опциональные ссылки на контекст:
    - `plan_id`, `section_id`, `unit_id`, `question_id`;
  - `role` — `user / assistant / system`;
  - `content` — текст сообщения;
  - `created_at`.

Решение: **чат работает в разных режимах контекста**:

- глобальный (`plan_id` пустой) — общие вопросы;
- по плану (`plan_id`) — вопросы про курс в целом;
- по уроку (`unit_id`) — вопросы по текущей теории и заданиям;
- по конкретному вопросу (`question_id`) — разбор конкретной задачи.

Бэкенд по этим FK поднимает:

- теорию (`plans_unit.theory`);
- вопросы и варианты (`plans_question`, `plans_choice`);
- ответы пользователя и фидбэк (`learning_answer`, `feedback_text`);
- статистику успехов по теме (`learning_question_stats`, `learning_unit_progress`).

Эти данные собираются в промпт для LLM, благодаря чему чат:

- отвечает по реальному контенту курса (минимизация галлюцинаций);
- может опираться на историю ошибок пользователя.

### 6. Использование LLM и RAG в архитектуре

LLM **не заменяет бизнес‑логику и БД**, а встроена в несколько чётких точек:

1. **Генерация курса (инкрементальная)**:
  - на вход: чанки исходного документа (найденные через векторный поиск), список тем, уровень;
  - на выход: сначала общая структура плана (список разделов и юнитов), затем содержание по каждому разделу/юниту;
  - бэкенд сохраняет:
    - каркас плана (`Plan → Section → Unit`) сразу, давая пользователю увидеть структуру;
    - наполнение (`theory`, `Question`, `Choice`) по мере готовности отдельных разделов/юнитов;
  - статусы `generation_status` на уровнях `plans_plan`, `plans_section`, `plans_unit` позволяют фронтенду:
    - показывать пользовательский интерфейс без ожидания полной генерации курса;
    - отображать прогресс генерации по разделам (например, «Раздел 1 готов, Раздел 2 в процессе»).
2. **Фидбэк по ответу**:
  - на вход: текст вопроса, правильный ответ, ответ пользователя, теория юнита;
  - на выход: `feedback_text`, который пишется в `learning_answer`.
3. **Аналитика слабых мест**:
  - на вход: агрегированные данные (`learning_section_progress`, `learning_question_stats`);
  - на выход: текстовое резюме слабых тем и рекомендации по повторению.
4. **AI‑чат**:
  - на вход: сообщение пользователя и контекст (курс/раздел/урок/вопрос + история чата);
  - на выход: ответ, который записывается в `ai_chat_message` и отдаётся на фронт.

Принцип: **все важные факты (ответы, прогресс, связи с контентом) живут в БД**,  
LLM — только трансформатор текста, работающий поверх этих данных.

---

### 7. Итог архитектурного решения

- Разделение на слои контента, действий и аналитики делает систему расширяемой:
  - можно добавлять новые типы вопросов и метрики, не ломая текущие данные;
  - легко масштабировать под несколько пользователей и публичные курсы.
- Связь с RAG реализована через метаданные во векторном хранилище, а не через жёсткую привязку в схеме БД.
- AI‑чат встроен в систему как **репетитор, работающий строго в контексте курса**, а не как «магический чёрный ящик».
- Хранение сырых ответов и фидбэка позволяет:
  - объяснять пользователю его ошибки;
  - строить отчёты по слабым местам;
  - доказывать корректность работы системы на защите диплома.
